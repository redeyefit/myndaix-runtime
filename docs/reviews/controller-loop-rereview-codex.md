**BLOCKER** `src/runtime/ledger/postgres_store.py:921`: `review_delivered()` is not a delivery signal. It advances on any `done` job with matching `repo_id/base_ref`, but `mxr` marks jobs done before `play-review.sh` writes the final inbox verdict at `orchestrator/play-review.sh:285` / `:298`. This can advance after only `kilabz` succeeds, after triage/delivery aborts, or from an unrelated/manual review sharing the same SHA. Fix: record a workflow-level delivery row only after final `deliver` succeeds, keyed by `repo_id + ref + head + base + dispatch_id`, and have the controller advance only on that.

**BLOCKER** `orchestrator/play-review.sh:140`: controller git calls are hardened, but the controller-launched review path is not. `_review_env()` at `src/runtime/controller.py:123` does not set `GIT_CONFIG_NOSYSTEM/GIT_CONFIG_GLOBAL`, and `confirm_pushed` runs plain `git ls-remote`. A repo/global `url.*.insteadOf` plus `protocol.ext.allow=always` can rewrite the already-validated URL into `ext::...`. Fix: pass the same hardened git env into play-review and use `git -c protocol.ext.allow=never -c protocol.fd.allow=never -c credential.helper= ...` for `ls-remote`.

**MAJOR** `src/runtime/controller.py:193`: lock heartbeat does not refresh the mtime being checked. `_stamp_lock()` rewrites `LOCK/meta`, but stale detection reads the directory mtime at `:205`; modifying an existing file does not update the parent directory. Also `release_lock()` at `:232` blindly deletes `LOCK`, so a paused old owner can delete a new owner’s lock after stale-steal. Fix: `os.utime(LOCK, None)` on heartbeat and release only when an ownership token still matches.

**MAJOR** `src/runtime/controller.py:371`: `claim_dispatch()` wins before `trigger_review()`. If the trusted script is missing, exits nonzero, or times out, the cursor is left falsely `dispatching` until stale reclaim, and can eventually block a head that was never dispatched. Fix: add a CAS `release_dispatch(repo_id, ref, head)` for definite front failures, or split claim into `triggering -> dispatched`.

**MAJOR** `src/runtime/controller.py:334`: the pending head is not pinned. Fetching a newer force-pushed head overwrites `refs/myndaix/controller/<slug>` while the old pending SHA is still in flight; if GC prunes it, `advance_cursor()` at `:324` can move to an unpinnable `reviewed_sha` because `_pin()` at `:309` ignores failure. Fix: pin pending heads under a separate ref before dispatch, verify `update-ref` succeeds before advancing, then move/delete refs atomically.

Autofix disable itself looks correct. `GIT_CONFIG_GLOBAL=/dev/null` will break production setups that rely on global `insteadOf`; use direct allowed remotes or an explicit allowlisted controller config. I did not run tests because this sandbox is read-only.
